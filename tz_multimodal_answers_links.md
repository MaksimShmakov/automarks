# Техническое задание

## Наименование задачи

Реализовать API-метод для обработки мультимодальных пользовательских запросов по ссылкам на внешние файлы и сформировать документацию для фронтенда по его использованию.

## Статус документа

Документ оформлен задним числом по фактически реализованному результату, зафиксированному в файле `result.md`.

## Цель

Обеспечить фронтенду единый backend-метод, через который пользователь может:

- отправить текстовый вопрос;
- приложить ссылки на изображения, PDF и аудиофайлы;
- получить единый текстовый ответ модели с учетом содержимого файлов, истории диалога и выбранной предметной базы.

## Бизнес-ценность

- Снизить требования к фронтенду: фронтенд передает только текст, ссылки и предметный контекст.
- Поддержать мультимодальные сценарии без прямой загрузки файлов в API.
- Дать возможность использовать один и тот же сценарий для разных предметов через `vector_keys`.
- Сохранить непрерывность диалога за счет серверной памяти по `user_id`.

## Область работ

Необходимо реализовать:

- новый endpoint `POST /api/v1/school/multimodal_answers_links`;
- авторизацию по API-ключу через заголовок `Authorization`;
- прием текстового сообщения и/или массива ссылок на файлы;
- загрузку файлов по URL на стороне сервера;
- определение типа файла по расширению URL;
- обработку документов и изображений через распознавание содержимого;
- обработку аудиофайлов через транскрибацию в текст;
- объединение распознанного контента с `user_message`;
- передачу итогового контекста в модель с учетом RAG и истории переписки;
- возврат ответа в формате JSON;
- поддержку сброса контекста через `POST /api/v1/school/reset` для `kind = "dialog_yarik_links"`;
- подготовку внешней инструкции для фронтенда по использованию метода.

## Функциональные требования

### 1. Endpoint мультимодального запроса

Сервис должен предоставить endpoint:

```http
POST /api/v1/school/multimodal_answers_links
Content-Type: application/json
Authorization: <PLATFORM_API_KEY>
```

### 2. Формат входных данных

Тело запроса должно поддерживать поля:

| Поле | Тип | Обязательность | Требование |
|---|---|---|---|
| `user_id` | `string` | да | Идентификатор пользователя для привязки истории диалога |
| `vector_keys` | `string[]` | да | Непустой массив ключей предметных баз |
| `kind` | `string` | да | Допустимое значение: строго `"dialog_yarik_links"` |
| `user_message` | `string` | нет | Текстовый вопрос пользователя |
| `links` | `string[]` | нет | Массив URL на поддерживаемые файлы |

Дополнительные правила валидации:

- хотя бы одно из полей `user_message` или `links` должно быть заполнено;
- `vector_keys` должен быть непустым массивом;
- неподдерживаемый `kind` должен приводить к ошибке валидации;
- все ссылки должны быть валидными и доступными для скачивания сервером.

### 3. Поддерживаемые типы файлов

Тип файла должен определяться по расширению URL.

Поддерживаемые расширения:

- документы и изображения: `.jpg`, `.jpeg`, `.png`, `.pdf`;
- аудио: `.ogg`, `.mp3`, `.wav`, `.m4a`, `.aac`, `.flac`, `.webm`, `.opus`.

Если тип файла не удается определить по расширению, сервис должен вернуть ошибку `400`.

### 4. Серверная обработка ссылок

Для каждого URL из `links` сервис должен:

- скачать файл по ссылке;
- применить таймаут скачивания не более 30 секунд на файл;
- определить тип файла;
- направить документы, изображения и PDF в контур распознавания содержимого;
- направить аудиофайлы в контур транскрибации;
- преобразовать результат обработки в текстовый контекст для модели.

### 5. Формирование запроса в модель

Перед вызовом модели сервис должен:

- объединить `user_message` и текст, полученный из файлов;
- добавить контекст истории диалога по `user_id`;
- использовать предметные базы, указанные в `vector_keys`;
- передать собранный контекст в модель для генерации ответа.

### 6. Формат ответа

При успешной обработке сервис должен возвращать:

```json
{
  "answer": "..."
}
```

Требования к полю `answer`:

- тип: `string`;
- содержит итоговый текстовый ответ модели;
- допускает наличие математических формул в формате LaTeX;
- не требует дополнительной вложенной структуры.

### 7. Память диалога

Сервис должен автоматически хранить историю переписки для каждого `user_id`.

Требования:

- использовать скользящее окно последних 10 сообщений диалога;
- сохранять контекст между последовательными запросами;
- при смене `vector_keys` сохранять историю пользователя и переключать предметную базу без очистки контекста.

### 8. Сброс контекста

Сервис должен поддерживать endpoint:

```http
POST /api/v1/school/reset
```

Тело запроса:

```json
{
  "user_id": "user_42",
  "kind": "dialog_yarik_links"
}
```

Ожидаемый ответ:

```json
{ "response": true }
```

## Требования к `vector_keys`

Формат ключа: `<предмет>_<класс>`, где класс находится в диапазоне от `05` до `11`.

Поддерживаемые префиксы предметов:

- `math`
- `mathprof`
- `physics`
- `chemistry`
- `society`
- `biology`
- `history`
- `russian`
- `literature`
- `english`
- `informatics`
- `geography`

## Нефункциональные требования

- API должен работать по JSON-контракту.
- Авторизация должна использовать существующий платформенный API-ключ без префикса `Bearer`.
- Ошибки валидации должны возвращаться в человекочитаемом виде через поле `detail`.
- Ответ должен быть пригоден для прямого отображения во фронтенде.
- Формулы в ответе должны быть совместимы с рендером через KaTeX или MathJax.

## Обработка ошибок

Сервис должен поддерживать следующие базовые сценарии ошибок:

| HTTP-код | Условие |
|---|---|
| `400` | неверный `kind`, пустой `vector_keys`, отсутствует и `user_message`, и `links`, ссылка недоступна, тип файла не поддерживается |
| `401` | неверный или отсутствующий API-ключ |
| `500` | внутренняя ошибка сервера |

Примеры ожидаемых сообщений:

- `Unsupported dialog kind for /v1/school/multimodal_answers_links: ...`
- `No input provided: send at least one of (links, user_message).`
- `vector_keys must be non-empty`
- сообщение о недействительной ссылке;
- сообщение о невозможности определить тип файла по расширению.

## Критерии приемки

Задача считается выполненной, если:

1. Endpoint `POST /api/v1/school/multimodal_answers_links` принимает запрос только с текстом и возвращает ответ `200`.
2. Endpoint принимает запрос только со ссылками и возвращает ответ `200`.
3. Endpoint принимает комбинированный запрос с текстом и ссылками и возвращает ответ `200`.
4. При передаче изображений, PDF и аудио содержимое файлов учитывается в ответе модели.
5. При неверном `kind` возвращается `400`.
6. При пустом `vector_keys` возвращается `400`.
7. При отсутствии одновременно `user_message` и `links` возвращается `400`.
8. При недоступной ссылке возвращается `400`.
9. При неподдерживаемом расширении файла возвращается `400`.
10. При неверном API-ключе возвращается `401`.
11. История диалога сохраняется по `user_id` между последовательными запросами.
12. Endpoint `POST /api/v1/school/reset` очищает контекст и возвращает `{ "response": true }`.
13. Для фронтенда подготовлена отдельная инструкция с примерами запросов и ответов.

- документация для фронтенда по контракту API;
- описание поддерживаемых типов файлов, ошибок и примеров интеграции.
