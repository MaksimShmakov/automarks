# Playbook по проведению экспериментов

## Когда стоит запускать эксперимент
Запускаем эксперимент, если:
- есть конкретная гипотеза, а не просто идея "давайте попробуем";
- понятно, на какую метрику влияем;
- можно разделить трафик между вариантами;
- у команды есть критерий остановки и критерий успеха.

## Минимальный шаблон хорошей гипотезы
Формула:
Если мы [изменение], то [метрика] изменится на [ожидаемый эффект], потому что [инсайт].

Пример:
Если мы упростим первый экран бота, то CR в переход к следующему шагу вырастет на 8%, потому что пользователю будет проще понять первый CTA.

## Как оформить эксперимент в платформе
1. Создайте карточку эксперимента.
2. Укажите название так, чтобы через месяц было понятно, что именно тестировали.
3. Включите A/B, если гипотеза реально требует сплита трафика.
4. Выберите ветку для API, если тест должен раздаваться по коду ветки.
5. Заполните:
   - варианты;
   - метрику влияния;
   - ожидаемое изменение;
   - гипотезу;
   - объем трафика;
   - длительность.
6. Переведите карточку из backlog в draft, затем запустите тест.

## Как выбрать сплит трафика
- 50/50: дефолт для большинства тестов.
- 70/30: когда нужно снизить риск для основного варианта.
- Свой сплит: только если есть понятная причина и это заранее согласовано.

## Как вести эксперимент во время прогона
- Не меняйте гипотезу на лету.
- Не сравнивайте результаты по ощущениям, только по заранее выбранной метрике.
- Не останавливайте тест слишком рано из-за первых колебаний.
- Если условия теста изменились, лучше зафиксировать это в комментарии и отправить гипотезу на ретест.

## Как завершать эксперимент
Теперь итог можно внести прямо из карточки:
1. Нажмите кнопку `Завершить`.
2. В попапе выберите итог: `Успех`, `Провал` или `Ретест`.
3. Заполните даты старта и окончания.
4. Добавьте ссылку на dashboard, если нужен конкретный отчет.
5. Внесите результаты по варианту A и B.
6. При необходимости оставьте комментарий с выводом.
7. Сохраните итог.

После этого карточка уходит в библиотеку гипотез.

## Как интерпретировать результаты
- `Успех`: вариант дал ожидаемый или полезный эффект, гипотезу можно масштабировать.
- `Провал`: эффект не подтвердился или ухудшил метрику.
- `Ретест`: данных недостаточно, реализация была нестабильной или нужны другие условия запуска.

## Что обязательно должно быть в комментарии
Если комментарий заполняется, в нем лучше коротко зафиксировать:
- что победило;
- по какой метрике приняли решение;
- есть ли ограничения или оговорки;
- что делать дальше: масштабировать, закрыть или ретестить.

## Типовые ошибки
- Гипотеза сформулирована слишком размыто.
- Нет одной главной метрики.
- Трафик не соответствует запланированному сплиту.
- Итоги не занесены сразу после завершения.
- В комментарии нет решения, только набор цифр.

## Короткий чеклист перед запуском
- Гипотеза сформулирована.
- Метрика выбрана.
- Сплит согласован.
- Ветка выбрана, если тест идет через API.
- Понятно, когда тест считается завершенным.
- Есть ссылка на dashboard или понятный источник цифр.

## Короткий чеклист перед финализацией
- Даты старта и окончания заполнены.
- Результаты по A и B собраны.
- Итоговый статус выбран осознанно.
- Комментарий отвечает на вопрос "что решили делать дальше".
