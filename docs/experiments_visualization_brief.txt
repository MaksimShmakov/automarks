2. Цели и задачи проекта
Основная цель: сделать понятный дашборд по экспериментам в DataLens, чтобы на одном экране было сразу видно, что тестируется сейчас, что уже завершилось и какие выводы можно сделать.
Что должно быть понятно пользователю за 30-60 секунд:
- какие эксперименты идут прямо сейчас;
- какие эксперименты завершились успешно, неуспешно или ушли на повторную проверку;
- по каким типам гипотез чаще бывают хорошие и плохие результаты.

Какие задачи должны быть решены:
1. Сделать верхний блок с 5 главными показателями:
- всего экспериментов;
- активных сейчас;
- успешных;
- неуспешных;
- отправленных на повторную проверку.

2. Сделать блок "Активные эксперименты".
В этом блоке по каждому эксперименту должны быть видны:
- название эксперимента;
- текущий статус;
- бот;
- ветка или сценарий;
- основная метрика;
- дата запуска;
- ответственный;
- ссылка на источник по эксперименту или связанный отчет.

3. Сделать блок "Завершенные эксперименты".
В этом блоке по каждому эксперименту должны быть видны:
- дата завершения;
- название эксперимента;
- бот;
- ветка или сценарий;
- основная метрика;
- как был поделен трафик между вариантами;
- итог эксперимента;
- короткий вывод по результату.

4. Сделать график запусков.
На графике должно быть видно, сколько экспериментов запускали по неделям или по месяцам.

5. Сделать воронку по статусам.
Статусы:
- backlog;
- draft;
- в тесте;
- на оценке;
- успех;
- провал;
- ретест.

6. Сделать отдельный срез по типам гипотез.
Нужные группы:
- стартовый экран;
- сегментация;
- номер;
- подписка;
- дожимы;
- продажа;
- другое.

7. Для A/B-тестов показать результаты двух вариантов отдельно:
- вариант A;
- вариант B;
- итог по выбранной метрике.

8. Добавить фильтры:
- период запуска;
- период завершения;
- статус;
- бот;
- ветка;
- есть ли A/B-тест;
- тип гипотезы;
- метрика;
- ответственный.

9. Дать возможность перейти из строки эксперимента к источнику по этому эксперименту или к связанному отчету.

10. Добавить экспорт списка экспериментов в таблицу.

11. Дашборд должен быть понятен человеку без технической подготовки и без отдельного обучения.

Что считаем готовым результатом:
- за 30-60 секунд можно понять, какие тесты активны и чем закончились последние завершенные эксперименты;
- можно отфильтровать данные по боту, ветке и периоду;
- можно открыть источник результата по ссылке из эксперимента;
- экран можно использовать как рабочий архив завершенных гипотез.

3. Целевая аудитория
Кто будет использовать дашборд:
- маркетологи;
- аналитики;
- тимлиды;
- руководители.
Какой уровень технической подготовки у пользователей:
- базовый;
- интерфейс должен быть понятен без специальных знаний.

4. Источники данных
- Основной источник данных: текущая база экспериментов.
- Отдельную ручную загрузку данных делать не нужно.
- Данные, которые уже есть и должны использоваться:
- название эксперимента;
- статус;
- бот;
- ветка;
- наличие A/B-теста;
- варианты гипотезы;
- метрика влияния;
- ожидаемый эффект;
- текст гипотезы;
- объем трафика;
- длительность;
- даты старта и завершения;
- ссылка на связанный дашборд;
- результаты по вариантам A и B;
- комментарий;
- автор;
- дата обновления.

5. Ожидаемые метрики и KPI
Какие метрики должны быть на дашборде:
- всего экспериментов;
- активных экспериментов;
- успешных экспериментов;
- неуспешных экспериментов;
- экспериментов на ретесте;
- количество запусков за период;
- средняя длительность эксперимента;
- доля гипотез, по которым уже принято финальное решение;
- частота успехов по типам гипотез;
- частота ретестов по ботам и веткам;
- сравнение результатов варианта A и варианта B.

Какую структуру данных нужно визуализировать:
- одна запись по одному эксперименту;
- статус эксперимента;
- тип гипотезы;
- даты;
- метрика;
- результат;
- ответственный;
- ссылка на источник цифр.

6. Дизайн и пользовательский интерфейс
- Интерфейс должен быть простым и спокойным, без перегруженных элементов.
- Главный акцент на понятных подписях и удобном чтении таблиц.
- Названия блоков и полей должны быть написаны простым языком.
- Основной сценарий: пользователь открывает экран и быстро понимает общую картину.
Нужно учитывать удобство использования:
- читаемые шрифты;
- достаточный контраст;
- логичный порядок блоков;
- отсутствие лишних действий для перехода к деталям.

7. Визуализация данных
Предпочтительные форматы:
- блоки с главными числами;
- таблицы;
- столбчатые графики;
- линейные графики;
- воронка по статусам.

Что не нужно делать в первой версии:
- сложные визуализации, которые трудно читать без пояснений;
- перегруженные экраны с большим количеством мелких блоков.

Примеры и ориентиры:
- можно опираться на текущую структуру полей, статусов и результатов по экспериментам;
- визуально важнее простота и читаемость, чем декоративность.

8. Функциональные возможности
Какие функции должны быть в дашборде:
- фильтры;
- переход к источнику по эксперименту;
- переход в связанный отчет;
- экспорт таблицы.

Нужна ли интеграция с другими системами:
- для первой версии нет;
- используются уже существующие данные.

Дашборд должен быть автоматическим или ручным:
- автоматическим;
- отдельная ручная подготовка данных вне платформы не нужна.

Дополнительные требования:
- дашборд должен быстро работать даже при большом количестве экспериментов;
- дашборд должен быть пригоден для ежедневной работы команды.

11. Другие пожелания и комментарии
- Во второй версии желательно добавить отдельное поле "итог" или "решение", чтобы не искать вывод в свободном комментарии.
- Во второй версии желательно добавить единое поле "основная метрика", чтобы было проще фильтровать все эксперименты одинаково.
